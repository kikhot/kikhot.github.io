Redis 的 Zset（Sorted Set，有序集合）是一个很适合做“排序 + 唯一成员”的数据结构。它看起来像 Set，但每个成员都会额外绑定一个 `score`，Redis 会按 `score` 维护顺序。

这篇笔记参考了腾讯云开发者社区文章 [《Redis数据结构：Zset类型全面解析》](https://cloud.tencent.com/developer/article/2343216)，并按我自己的理解重新整理，方便后续复习和查阅。

## Zset 解决什么问题

Zset 适合这种数据模型：

```text
member -> score
```

其中：

- `member` 是唯一的，同一个 Zset 中不会出现重复成员。
- `score` 是排序权重，Redis 会根据它维护有序结果。
- 当成员的 `score` 更新后，排序位置也会随之变化。

如果只需要去重，用 Set 就够了；如果既要去重，又要根据分数、时间、优先级排序，Zset 就很顺手。

## 常见应用场景

### 排行榜

把用户 ID 作为 `member`，积分、分数、热度作为 `score`。

常见查询包括：

- 查询 Top N 用户。
- 查询某个用户排名。
- 查询某个分数区间内的用户。
- 更新用户分数并自动调整排名。

### 时间线

把消息 ID、文章 ID 或动态 ID 作为 `member`，发布时间戳作为 `score`。

这样可以按时间顺序拉取内容，例如：

- 最新动态。
- 某个时间段内的内容。
- 某个用户之后的增量内容。

### 延时队列

把任务 ID 作为 `member`，任务应该执行的时间戳作为 `score`。

消费者定期取出 `score <= 当前时间` 的任务并执行。这个方案简单直观，但要注意并发消费、重复执行和任务失败重试问题。

### 优先级队列

把任务作为 `member`，优先级作为 `score`。分数越高或越低优先执行，取决于业务约定。

## 底层结构怎么理解

Zset 的核心诉求是：

1. 能快速按成员查找分数。
2. 能按分数保持有序。
3. 能支持范围查询和排名查询。

因此它不是单纯用一个数组或链表解决，而是根据数据规模和编码策略选择更合适的底层结构。

## 小数据量：紧凑编码

在数据量较小、成员长度较短时，Redis 会倾向使用更节省内存的紧凑结构。

这类结构的特点是：

- 内存占用低。
- 数据连续存储，对小集合很友好。
- 插入或更新时可能需要移动较多数据。

所以它适合“小而短”的 Zset，不适合频繁修改的大集合。

## 大数据量：跳表

当 Zset 规模变大后，Redis 会使用跳表等结构来维护有序性。

跳表可以理解为“带多级索引的有序链表”：

- 最底层保存完整有序链表。
- 上层保存稀疏索引。
- 查找时从高层索引开始跳跃，逐层向下缩小范围。

它的优势是插入、删除、查找都能保持比较稳定的效率，适合内存数据库中频繁更新的有序集合。

## 为什么 Redis 更适合跳表，而不是 B+ 树

B+ 树更常出现在数据库索引里，因为它非常适合磁盘页读写和范围查询。

Redis 的 Zset 主要运行在内存里，关注点不一样：

- 内存访问没有磁盘 I/O 页读取的瓶颈。
- 跳表实现相对简单，插入删除不需要复杂的节点分裂与合并。
- Zset 经常面对实时更新排名的场景，跳表比较适合这种写入模型。

所以不能简单说跳表比 B+ 树更好，而是它们适合的系统约束不同。

## 常用命令速记

### 添加或更新成员

```bash
ZADD ranking 100 user:1
ZADD ranking 80 user:2 120 user:3
```

如果成员已存在，`ZADD` 会更新它的 `score`。

### 查询成员分数

```bash
ZSCORE ranking user:1
```

返回指定成员的分数。

### 查询排名

```bash
ZRANK ranking user:1
ZREVRANK ranking user:1
```

- `ZRANK`：按分数从小到大排名。
- `ZREVRANK`：按分数从大到小排名。

### 查询范围

```bash
ZRANGE ranking 0 9 WITHSCORES
ZREVRANGE ranking 0 9 WITHSCORES
```

适合做排行榜 Top N。

### 删除成员

```bash
ZREM ranking user:2
```

### 按分数统计

```bash
ZCOUNT ranking 80 120
```

统计指定分数区间内的成员数量。

## 使用时的注意点

### score 要有清晰含义

`score` 可以表示积分、时间戳、优先级、热度，但最好在业务里保持单一含义。不要让一个 Zset 同时承担多种排序规则。

### member 不要太大

Zset 的成员通常建议存 ID，而不是完整 JSON。详细内容放到 Hash、String 或数据库里，通过 ID 再查询。

### 延时队列要处理并发

如果多个消费者同时扫描同一个 Zset，需要用原子操作、Lua 脚本或分布式锁避免重复消费。

### 排行榜要考虑冷热数据

如果排行榜数据非常大，可以考虑按时间窗口拆分，例如：

- `ranking:daily:2026-07-02`
- `ranking:weekly:2026-W27`
- `ranking:global`

这样查询和清理都会更容易。

## 我的理解

Zset 最值得记住的一句话是：

> 它是“唯一成员 + 可更新权重 + 有序访问”的组合。

当需求里同时出现“排名、范围、时间线、优先级、延迟执行”这些关键词时，就应该先想想 Zset 是否合适。

不过 Zset 不是万能队列，也不是复杂搜索引擎。它适合解决清晰的排序问题；如果排序规则很多、过滤条件复杂，可能需要结合数据库、搜索引擎或专门的任务队列一起设计。
